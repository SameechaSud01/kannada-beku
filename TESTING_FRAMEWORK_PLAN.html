Where we are today

The foundation is already real, and it's healthy: 14 suites, 102 tests, all passing in ~0.5 s. What's missing is everything above the unit layer — nothing renders a component, nothing exercises the offline sync queue end-to-end, and nothing drives the actual app.

Phase 1 Coverage + content integrity tests

Two cheap wins that pay off immediately.

Coverage with a ratchet, not a wall

Turn on coverage collection scoped to the pure-logic folders where 100% of the current tests live, and set thresholds slightly below today's numbers so the build fails only when coverage drops. No retrofitting tests to hit an arbitrary number.

// package.json → "jest"
"collectCoverageFrom": [
  "lib/**/*.ts", "stores/**/*.ts", "utils/**/*.ts",
  "hooks/**/*.ts", "src/**/*.ts",
  "!**/*.d.ts"
],
"coverageThreshold": { "global": { /* set from the Phase-1 baseline run */ } }

Plus two scripts: test:watch and test:coverage.

Content integrity tests — the highest-value suite this app can have

The lessons in lessonContent.ts are the product, they're canonical in TypeScript (not the DB), and content bugs have bitten before (the iḷḷi→illi fix, the ISO diacritic normalization pass). A schema suite makes every future content edit self-checking:

  • Every lesson/sub-section/item has a unique, stable ID; no duplicates across lessons.
  • Transliterations conform to the ISO diacritic conventions adopted 2026-06-26 (no banned characters, no stray ASCII approximations).
  • Every referenced audio asset actually exists in the bundle; no orphaned audio files.
  • Kannada fields contain Kannada script; English fields don't; no empty strings.
  • Game content (quiz options, opposites pairs, dictation aksharas) is well-formed — e.g. the correct answer is always among the options.

Phase 2 Component tests with React Native Testing Library

RNTL is installed and unused. The setup work is the real cost here — Reanimated v4 and gesture-handler need their Jest mocks wired in jest.setup.js before anything that touches an animated component will render. Once that's done, tests target the shared building blocks, not whole screens:

  • System-state components (components/states/): loading, failure, empty each render their message and retry action.
  • Answer feedback: correct → green treatment, wrong → red, per the locked owner override.
  • Lesson runner steps: given a step, the right variant renders and "continue" fires.
  • Accessibility as assertion: query by role/label (per CLAUDE.md), which makes missing accessibilityLabels test failures for free.
Not doing: full-screen snapshot tests — CLAUDE.md rules them out as brittle, and they'd fight every visual polish pass you do.

Phase 3 Integration tests: sync queue and progress flows

This is where the scariest bugs live — the ones the manual plan's Suites M/N (offline & multi-device) exist for. The approach: a shared fake Supabase client in __tests__/testUtils/ (an in-memory table + controllable failure/latency switches, injected via jest.mock of lib/supabase), then test the real stores against it:

  • Complete a game offline → writes queue locally → reconnect → queue flushes once, in order, no duplicates.
  • Flush fails mid-way → queue retains the unflushed tail; nothing is dropped.
  • End-of-game mastery refresh recomputes the client-side rollup and Home's rings see it.
  • Sign out / sign in as a different user → per-user state fully resets (the AUTH-08c leak scenario).
  • Lesson completion unlocks the next lesson without restart (the LRN-03 regression).

Each of these encodes a bug class this project has actually had, and they run in milliseconds with no simulator.

Phase 4 End-to-end flows with Maestro

Recommendation: Maestro, not Detox. Maestro drives a dev build on the simulator from plain YAML, needs no native test harness compiled into the app, and survives Expo SDK upgrades — Detox with RN 0.81/new-architecture is a maintenance tax this one-person project shouldn't pay. Flows live in .maestro/ and map 1:1 to the critical IDs in the manual plan:

FlowCovers manual IDs
Fresh install → onboarding → lands on Home; relaunch doesn't re-loopONB-01, ONB-06
Email login; force-quit; relaunch still authenticatedAUTH-02, AUTH-07
Complete a lesson → next lesson unlocksLRN-03
Play one game to completion → rings/mastery advance on HomeHOME-03, F-suite
Airplane mode mid-game → finish → reconnect → progress syncedM-suite
# .maestro/lesson-unlock.yaml
appId: com.kannadabeku.app
---
- launchApp
- tapOn: "Learn"
- tapOn: "Greetings"
- runFlow: helpers/complete-lesson.yaml
- assertVisible: "Names"        # next lesson now unlocked

What E2E deliberately won't cover: OAuth (Google/Apple system sheets aren't automatable), real audio output, and the physical-device matrix — those stay in the manual plan, which shrinks to the things only a human can check.

Decision needed: the auth flows need a strategy — a dedicated Supabase test account (simplest; I'd seed one and keep its creds in .env, never committed), or a dev-only auth bypass. I'd start with the test account.

Phase 5 Wire it into CI and write it down

  • PR pipeline (unchanged runtime): swap npm test -- --ci for the coverage run so the ratchet gates merges. Unit + component + integration all run here — they're fast.
  • E2E cadence: not on every PR. Maestro needs a macOS runner (~10× the per-minute cost of Linux) and a built app. Run it nightly or on-demand via workflow_dispatch; a later option is EAS Workflows' hosted Maestro job.
  • Pre-push hook: already runs Jest; it picks up the new suites automatically. No change.
  • docs/TESTING.md: one page — the layers, how to run each, where fakes live, when to add which kind of test, and the split between automated coverage and the manual plan.

What I'm deliberately leaving out

  • Database-level tests — migrations are run by hand in the Supabase dashboard and the live DB has known drift from the repo's SQL; there's no CLI-managed shadow DB to test against. Not worth faking.
  • Visual regression / screenshot diffing — high flake, low payoff while the UI is still being actively redesigned. Revisit post-launch if at all.
  • Storybook — a component workbench, not a test layer; adds a dependency surface without catching bugs the RNTL tests won't.

Confirm before I start

  1. Scope — all five phases, or trim? (Phases 1–3 are the core; 4 is the biggest lift and the biggest payoff.)
  2. E2E auth strategy — dedicated test account (my recommendation) vs. dev-only bypass.
  3. Hand-off points — I'll pause after each phase for you to review and run things yourself, per our usual rhythm.

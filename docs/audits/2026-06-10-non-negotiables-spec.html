How to use this document

This is a defensive contract, not a design doc. Every item below is something the app currently depends on. Most failures here are dangerous precisely because they are silent: no exception, no red screen — just wrong data, lost progress, or muted audio that surfaces only in production.

It is grounded in and subordinate to the locked foundation docs in docs/foundation/ (STATE.md, NAVIGATION.md, CONTENT.md, CONTRADICTIONS.md). Where the live code on main diverges from those docs, this audit reports the code as the source of truth and flags the divergence.

LOCKED decided; change needs a spec PR + owner sign-off SILENT breaks with no error thrown LOUD breaks visibly (crash/throw) OPEN undecided / not yet wired TARGET proposed threshold, not yet enforced
Reading guide: the Why blocks (gold) explain the rationale behind each invariant — delete the rationale and someone will reopen the decision. The red blocks are the silent-failure tripwires.

Core data models & their invariants

Three layers hold state: Zustand stores (client), the Supabase schema (server), and the API/RPC contracts that bridge them. Each has invariants that must never break.

1.1 — Zustand stores LOCKED

Three stores, one per concern, in stores/. useAuthStore is in-memory only; the other two persist to AsyncStorage.

StorePersisted?Storage key (literal)Source of truth
useAuthStoreNo (in-memory)n/a — Supabase client persists the session via expo-secure-storeSupabase session
useUserStoreYes · AsyncStorageuser_prefsServer public.users hydrates it on each auth event
useProgressStoreYes · AsyncStoragekannada-beku-progress (migrates legacy kannada-baa-progress)Client-first; completedLessons is a hydrated cache of server truth
INVARIANT — Storage keys are permanent. The literals user_prefs and kannada-beku-progress must never change. Renaming a key does not throw — it points the store at an empty slot, so every existing user silently loses all preferences and progress on their next launch. The legacy-key migration adapter in progressStore.ts (copies kannada-baa-progress → new key on first read, then deletes the old key) is one-shot and must stay until well past the rebrand cohort.

useProgressStore — the numeric invariants

Why idempotent: users frequently replay lessons. If completion incremented streak/XP on every replay, the metrics become meaningless. The streak answers "did you practice today?", not "how many times did you tap Done?" This contract is pinned by progressStore.test.ts (resolved C6) — those tests are themselves non-negotiable.

User-switch isolation — the cross-account firewall

AppGate compares session.user.id against useUserStore.userId on every auth change. First bind (stored id null) just records the id. A mismatch fires resetForUser(newId) + useProgressStore.reset(), and routing waits one tick for the reset to land.

INVARIANT — scope partitioning of resetForUser. It must clear per-user fields (hasCompletedOnboarding, displayName, learningMode, motivations, dailyGoalMinutes, dailyReminderTime, ttsRate, autoReplay, hasSeenBasicsGuide) and keep install-scoped flags (mode, hasSeenTtsWarning, hasSeenBasicsHomeNudge, permissionDenials, isHydrated). Mis-bucketing a field here is silent: clear too much and you re-prompt device-level things every login; clear too little and a new user inherits the previous user's onboarding state and skips onboarding, or sees their progress.

Hydration gate

Both useUserStore.isHydrated AND useProgressStore.isHydrated must be true (set in each store's onRehydrateStorage callback) before AppGate renders a routing decision.

Why gate first: the persisted onboarding flag chooses the redirect target. Route before hydration resolves and an onboarded user briefly flashes the login screen, or the routing loops. Removing either setHydrated(true) callback means routing never unblocks — a permanent splash hang. SILENT

1.2 — Supabase schema LOCKED

Client: supabase.ts — a singleton reading EXPO_PUBLIC_SUPABASE_URL / EXPO_PUBLIC_SUPABASE_ANON_KEY (throws at module load if absent). Three table families are read/written by app code today; the rest are scaffolded.

TableRoleLoad-bearing columns
usersIdentity + onboarding answers + per-user prefsid(PK→auth), email, name, learning_mode, motivations[], daily_goal_minutes, onboarding_completed_at, daily_reminder_time, tts_rate, auto_replay
lessonsReference rows + completion anchorid(uuid PK), lesson_no(unique, ordering), slug(unique — bridge to string IDs), content_json(jsonb: reference.words/phrases)
user_lesson_progress"Did I finish X?" + best score(user_id,lesson_id) composite PK, completed_at, score(0–100 check)

emergency_phrases · karnataka_fun_factsServer-backed content surfacesemergency: category∈{auto,trouble,basics}; facts: fact_no(unique)
INVARIANT — column names are an API. The client selects columns by literal name (e.g. USER_SELECT in users.ts). A Supabase column rename does not break the build — TypeScript can't see the DB — and PostgREST returns the remaining columns happily, so the field just arrives undefined. Renames of slug, lesson_no, content_json, completed_at, score, is_correct, sort_order, or any prefs column are silent data loss. SILENT

RLS is part of the contract

For the active tables: users = select/update own row only (insert via the on_auth_user_created trigger, never the client). lessons/*_items/emergency_phrases/karnataka_fun_facts = select for any authenticated user, no client writes (seed-only). *_progress = select/insert/update own, no delete — writes go only through the record_* RPCs.

Scaffolded tables are a tripwire. conversation_* and quick_quiz_* exist in the DB. Wiring any of them from a chat prompt without first applying the ulp_*_own RLS pattern is a security hole. Per STATE.md, each needs its own spec, accessor, RLS pass, and registered query/mutation keys before the first client read/write.

1.3 — API & RPC contracts LOCKED

All progress writes flow through SECURITY INVOKER RPCs with personal-best UPSERT semantics. The function names and parameter names are matched by string.

RPCParams (exact)Personal-best semantic
record_lesson_completionp_lesson_id uuid, p_score intscore = GREATEST(old,new); completed_at = COALESCE(old,new)
record_opposites_attempt
record_dictation_attempt
record_image_match_attempt		p_item_id uuid, p_is_correct boolis_correct OR-merges (once true, stays true); attempts++; last_played = now. Each fires the overall-progress recompute trigger.
recompute_overall_progressp_user_id uuid (trigger-driven, SECURITY DEFINER)See formula below
INVARIANT — RPC signatures. supabase.rpc('name', {p_x: ...}) resolves both the function name and the named args at runtime. Rename the SQL function or any p_* parameter and the call returns a PostgREST error the app mostly swallows into a "couldn't save" toast — progress silently stops persisting. SILENT / soft-LOUD

The overall-progress formula LOCKED

progress_pct = 50·(lessons_done/8) + (50/3)·(opposites_done/8) + (50/3)·(dictation_done/8) + (50/3)·(image_match_done/8)

  • lessons_done = completed lessons (completed_at IS NOT NULL), capped at 8.
  • A game is "done" for a lesson when ≥80% of that lesson's items have is_correct = true lifetime (correct/total >= 0.8), capped at 8 lessons.
  • The split is hard-coded: 50% lessons + 50% games over exactly 3 games.
Why it's frozen at 3 games: the formula is locked across client and DB (see memory note "overall-progress formula locked to 3 games"). Changing the 80% threshold, the 8-lesson cap, or the 3-way split silently re-bases every user's headline percentage — and there is no migration that recomputes historical rows unless triggered. Any change is a LOCKED-decision change, not a refactor.
Live consequence (C13): Image Match is hidden from Practice but still counts as 1 of the 3 games. With no UI path to accumulate image_match_progress, overall progress can never reach 100% while the game is hidden. Treat the cap as known until C13 is resolved.

2.1 — Authentication & session LOCKED

INVARIANT — chunked session storage. Supabase sessions exceed the iOS Keychain 2048-byte item limit, so supabase.ts wraps expo-secure-store in a ChunkedSecureStoreAdapter (1500-byte chunks + a _meta count key). Removing or "simplifying" this adapter back to a plain secure-store call means the session write silently truncates/fails and users are logged out on every cold start. SILENT
INVARIANT — reset-password exception. While the user is on (auth)/reset-password with an active recovery session, AppGate must early-return (no redirect). Remove that guard and the recovery session bounces the user into the app before they set a password — the reset flow becomes uncompletable. SILENT

2.2 — Lesson progression

Code-vs-doc divergence (report): The foundation docs describe a scenario→intake→drill→output→done machine with fill_blank drill types (C7). The live main runner is a fixed phase sequence — idle → situation → teach_words → practice_words → teach_phrases → practice_phrases → summary → real_world → done — with no drill-type dispatch. C7's fill_blank placeholder is effectively resolved-by-removal. The phase enum is the contract now; the doc-described drill engine is not present.
  • Unlock rule (linear): lesson N unlocks when completedLessons.length >= N-1. Lesson 1 always open. An off-by-one here silently locks or unlocks the wrong lesson.
  • Completion is server-first: useCompleteLessonMutation calls recordLessonCompletion(lessonId, 100) first, and only on success runs completeLesson() → updateStreak() → recordActivity(), then invalidates ['lesson-completions',userId] + ['overall-progress',userId].
  • Score is hard-coded to COMPLETION_SCORE = 100; minutesPracticed is a placeholder ESTIMATED_MIN_PER_LESSON = 5 until per-lesson timing is wired.
  • Slug bridge: the client persists/compares lesson slugs; the DB keys on UUID. fetchLessonIdBySlug (cached in lessons.ts) is the join. A lesson row missing its slug raises LessonNotRegisteredError → "not registered on server" — the one place this failure is loud.
Why server-first: if the store mutated before the DB write, a failed/retried write would let the local idempotency guard mark the lesson done while the server never recorded it — progress that survives reinstall on one device but vanishes on another. Server-first keeps the DB as the canonical completion ledger.

2.3 — Audio playback

MVP audio is device TTS via deviceTtsAudioService.ts (expo-speech). expo-av handles audio-mode globals, mic recording, and recorded-file playback.

Why a boot probe: Kannada TTS voices are not guaranteed installed on every device. The one-time TtsProbe (gated on hasSeenTtsWarning) surfaces a "install a voice" dialog rather than letting the whole app fail mutely. If the probe never runs (because userHydrated never flips), a device with no Kannada voice fails every play() with only a console warning.
Inconsistency to preserve intentionally: the Emergency screen logs audio failures silently; lesson phases surface a retry toast (Toasts.audioFailed). This is deliberate — don't "unify" them into duplicate toasts without checking the Emergency UX.

2.4 — State persistence & boot order

  1. SplashScreen.preventAutoHideAsync() at module scope.
  2. useFonts() loads 11 variants (DM Sans, Noto Sans Kannada, Baloo Tamma 2). Render is blocked — if (!fontsLoaded) return null — until they resolve.
  3. Fonts ready → SplashScreen.hideAsync() + Audio.setAudioModeAsync() + TTS probe (effects, non-blocking).
  4. Stores rehydrate from AsyncStorage → each sets isHydrated.
  5. AppGate waits for !authLoading && userHydrated && progressHydrated, runs the user-switch firewall, then routes with router.replace.
INVARIANT — Kannada is rendered by Noto Sans Kannada. Kannada glyphs use Fonts.notoSansKannada.*. If that family fails to load, or a teaching component reaches for Fonts.dmSans/Fonts.baloo for Kannada text, the script renders as tofu boxes (□□□) — no error, just unreadable lessons. Latin-only Baloo/DM Sans fonts do not cover the Kannada block. SILENT

Integration points

3.1 — Expo SDK & native runtime

  • Expo SDK 54 / RN 0.81.5 / React 19.1, newArchEnabled:true (Fabric/TurboModules). A version bump that drops New Architecture support, or mismatches Reanimated 4 / Worklets 0.5, breaks gesture & animation at runtime.
  • Expo Router 6, file-based. The route tree is the navigation contract: group folders (auth), (tabs), (games), onboarding drive AppGate's segments[] checks. Renaming a group folder silently changes which redirect rule matches.
  • Config plugins in app.json: expo-router, expo-secure-store, expo-font, expo-notifications, @react-native-google-signin (with the literal iosUrlScheme), expo-apple-authentication. The Google iosUrlScheme must stay in lockstep with EXPO_PUBLIC_GOOGLE_IOS_CLIENT_ID (it's the reversed form).
  • Deep-link scheme kannada-beku — the password-recovery link is the only deep-link handler. Changing the scheme without updating the Supabase allow-list silently kills password reset.
  • Bundle identifiers com.samsud01.kannadabeku (iOS + Android) anchor OAuth redirect URIs and push tokens — never change casually.

3.2 — expo-av & expo-speech

Covered as flow 2.3. The integration-level invariant: expo-speech is TTS-only; expo-av owns the audio-mode globals. They share the iOS audio session, so the setAudioModeAsync contract (silent-mode override on, recording off by default, no background audio) governs both. expo-av is on a 16.x line — its Audio.Sound/Audio.Recording API is slated to move to expo-audio/expo-video in a future SDK; a migration there must re-verify the silent-mode flag and the recording-mode reset.

3.3 — Google Stitch & Claude Design outputs

The contract: design artifacts are reference-only, never imported. The redesign handoff under spec_docs/Sameecha/playful_redesign_handoff/ contains .jsx and .html design references (Google Stitch / Claude Design exports: pl-home.jsx, pl-lesson.jsx, pl-auth.jsx, the Playful HTML mockups, etc.). No app code imports these files — they are visual specs that humans translate into the app's component system.

The translation must conform to the app's own styling contract, not the artifact's:

  • Inline styles + design tokens only. No NativeWind, no className=. Tokens live in constants/ (colors.ts, fonts.ts, spacing.ts, radius.ts). A Stitch/Claude export will arrive full of Tailwind classes or raw hex — porting those verbatim violates the locked DESIGN.md approach (this is exactly what bit the Opposites game in C1).
  • Token fidelity over pixel fidelity. Map exported colors to Colors.*, type to Fonts.*/TypeScale.*. Hard-coded values that bypass tokens silently drift from the brand system and break theming.
  • Kannada always renders via Noto Sans Kannada, regardless of what font a design export specifies (see 2.4).
  • The README still references NativeWind in two stale lines (C9) — that is documentation rot, not a dependency; do not reintroduce NativeWind on its authority.
Why reference-only: design tools optimize for a flat web mockup; the app is a New-Architecture RN tree with a token system, a custom floating TabBar, and Reanimated micro-interactions. Treating an export as importable code would smuggle web idioms (CSS classes, DOM elements, web fonts) into a native runtime where they silently no-op or mis-render.

3.4 — Supabase & TanStack Query

Covered in §1.2–1.3. Integration-level: the QueryClient is created in _layout.tsx; query/mutation keys are a locked convention (['lesson-completions',userId], ['overall-progress',userId], ['<game>-items',lessonNo], etc.). Mutations invalidate by exact key — a typo in an invalidation key is a silent stale-cache bug, not an error.

Performance & reliability thresholds

The codebase does not currently enforce numeric SLOs, so the values below are TARGET proposals grounded in how the boot path and audio path actually behave. Adopt them as the bar future changes must not regress.

MetricTargetGrounding / how to measure
Cold start to first interactiveTARGET < 2.5 s on iPhone SE (2nd gen)Gated by useFonts (11 variants) + store rehydrate. Measure splash-visible duration from preventAutoHideAsync to first routed paint.
Warm startTARGET < 1.0 sFonts cached by Expo; only hydration + auth probe remain.
Routing decision after hydrationTARGET < 1 frame visible flickerBoth stores hydrated before any router.replace; uses replace not navigate to keep the stack clean.
TTS audio latency (tap → speech)TARGET < 250 msSpeech.stop() + Speech.speak() are effectively synchronous to the OS engine; latency is dominated by the OS voice. No app-side debounce.
Recorded-file playback setupTARGET < 400 msAudio.Sound.createAsync unloads the prior sound first; measure create→playAsync.
Crash-free sessionsTARGET ≥ 99.5%No crash-reporter wired today — adopting one (e.g. Sentry) is a prerequisite to enforcing this. Until then this is aspirational.
Progress-write success rateTARGET ≥ 99% of completions persist server-sideServer-first mutation means a failed RPC shows a retry toast; track RPC error rate in Supabase logs.
Kannada-voice availability fallback100% of no-voice devices see the probe dialogReliability, not speed: the probe must fire once per install when !hasSeenTtsWarning.
Why these and not tighter numbers: the app is content-light (no large media bundles, TTS not recorded audio) and the dominant cost is font loading + first network round-trip for the user row. The honest gap is observability — there is no crash reporter or analytics SDK in package.json, so the crash-free and write-success targets cannot be verified yet. Wiring telemetry is the first reliability task, not picking a stricter percentage.
Reliability tripwire — fonts are a single point of failure for first paint. Because RootLayout returns null until all fonts load, a font that fails to fetch/cache (e.g. a removed @expo-google-fonts package or a corrupted asset) hangs the app on the splash screen with no timeout and no error UI. Consider a font-load timeout fallback before raising the startup SLO.

Future-audit checklist

Run this before merging anything that touches stores, the schema, auth, audio, or the boot path. Each box maps to an invariant above.

Data & persistence

  • Storage keys unchanged: user_prefs and kannada-beku-progress still literal; legacy migration adapter intact.
  • Idempotency tests pass: progressStore.test.ts still asserts no XP/streak double-count on replay.
  • resetForUser / reset scoping matches the per-user vs install-scoped table in §1.1 — no field moved buckets.
  • Both stores still call setHydrated(true) in onRehydrateStorage.
  • No new schema column rename without updating the matching *_SELECT / accessor and this doc.

API & server

  • RPC names + p_* params unchanged, or every call site updated in the same PR.
  • Overall-progress formula (80% / cap 8 / 50-50-over-3) untouched, or owner-signed as a LOCKED change with a recompute migration.
  • RLS applied before any scaffolded table (conversation_*, quick_quiz_*) gets its first client read/write.
  • New query/mutation keys follow the [resource, …scope] convention and invalidate correctly.
  • Chunked secure-store adapter still wraps the Supabase auth storage.

Flows & runtime

  • AppGate decision matrix intact, including the reset-password early-return and the user-switch one-tick wait.
  • Lesson completion is server-first (RPC before store mutation) and the unlock predicate is off-by-one-safe.
  • Phase enum changes update both the runner transitions and the app/lesson/[id].tsx switch.
  • Audio invariants: kn-IN, playsInSilentModeIOS:true, startsWith('kn') probe, Speech.stop() before speak, rate clamp [0.5,1.5], recording-mode reset.

Integrations & design

  • No NativeWind / className= reintroduced (grep returns zero outside node_modules).
  • Stitch/Claude exports ported to tokens — inline styles backed by constants/, Kannada on Noto Sans Kannada.
  • Google iosUrlScheme still matches EXPO_PUBLIC_GOOGLE_IOS_CLIENT_ID; deep-link scheme kannada-beku unchanged or Supabase allow-list updated.
  • expo-av / expo-speech SDK bump re-verifies silent-mode flag + recording reset (watch for the expo-audio migration).

Observability (the standing gap)

  • Crash reporter wired before claiming the ≥99.5% crash-free target as measured.
  • Cold-start & TTS-latency spot-checked on iPhone SE after any boot-path or font change.
  • Startup assertions: consider failing loud on missing storage keys / malformed user row / wrong-script font in dev.

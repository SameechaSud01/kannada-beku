Gamification, re-examined

Mapping Kannada Baa's current mechanics against Tim Gabe's 7 research-backed patterns · 2026-06-04

Short version: the app is already on the right side of this video's central thesis. It has deliberately avoided the "PBL theater" (no leaderboards, hidden XP, no badge tree) that the talk spends its first third dismantling. The biggest wins aren't in adding gamification — they're in surfacing two things the code already computes but never shows the user: daily-goal closure and evidence of competence.

Why this is an analysis, not a patch

Per .claude/CLAUDE.md, features can't be built from a chat prompt — each needs a per-feature spec doc in spec_docs/ that conforms to the docs/foundation/ locks. Several ideas below also brush against [LOCKED] decisions. So this is a prioritized menu; the recommended next step is drafting specs for the top 2–3, not editing components.

Pattern by pattern

1 · The PBL Fallacy — already won

Video: Points, badges and leaderboards are the scoreboard, not the game; they incentivize quantity over quality (LinkedIn, Foursquare walked them back).

In the code: Leaderboards and "streak shaming / leaderboard pressure" are a [LOCKED] non-goal in docs/foundation/SCOPE.md. xp is awarded in stores/progressStore.ts (20 pts ≥80%, else 10) but never rendered anywhere — grep finds zero XP UI. No badge / trophy / achievement system exists.

Verdict: Aligned by design. The instinct to keep XP invisible is exactly right. Don't "fix" it by surfacing a points counter. The only cleanup worth considering: XP is now near-vestigial (it feeds total_score server-side but drives no user-facing behavior) — flag it for a decision rather than quietly growing more logic around it.

2 · Engineering Winnability — the latent win

Video: Strava beats impossible global leaderboards with hyper-local segments — small, winnable competitions. The unit that's winnable for a solo learner is your own past self.

In the code: The data is already there. opposites_progress, dictation_progress, image_match_progress all store per-item is_correct / attempts with personal-best UPSERT semantics (the record_*_attempt RPCs in docs/foundation/STATE.md). But the game result screens (src/games/*/components/ResultScreen.tsx) show an ephemeral session score with no memory — "12" with no "your best was 9." The win is computed and thrown away.

Verdict: Latent — cheap to surface. "You beat your last run on this set" / "new personal best" is winnable, self-referential, and never shames. It needs the runner to read prior progress before the session and compare. Respects the locked no-leaderboard non-goal because the only opponent is yesterday-you.

3 · The S-Curve Problem — stay restrained

Video: More features ≠ more engagement; Habitica-style over-gamification causes cognitive overload and erodes the core utility.

In the code: The app is already disciplined. There's exactly one celebration stack (streak takeover → daily-goal dialog), and it has suppression logic: a streak milestone suppresses the goal dialog the same lesson (components/lesson/DoneCard.tsx, INTERACTIONS M4/M7). That's the right reflex.

Verdict: Aligned. This pattern is mostly a guardrail on the other six: when you add competence evidence (#7) and goal rings (#6), don't also add badges, levels, currencies and a shop. Every proposal below is additive to one surface, not a new system. If two ideas land, kill XP (#1) to keep net complexity flat.

4 · The Streak Trap — reframe, don't remove

Video: Streaks flip from motivation to obligation; increasingly flagged by regulators for fostering compulsive use.

In the code: Streak is [LOCKED] in docs/foundation/STATE.md and central to the loop — it's the first thing on home and profile. The intent is healthy: SCOPE explicitly bans "streak shaming," and the streak is defined as "did you practice today?" not "how many times did you tap done?" But the framing leans loss-averse in places: GoalCompleteDialog says the streak is "still alive" with a flame icon — that's the "don't lose it" lever the video warns about, not the "look how far you've come" lever.

Lock boundary

You cannot remove or replace the streak (locked), and shouldn't propose the opposite. What's in bounds and fully consistent with "warm and forgiving": copy and emphasis. Reframe milestone/goal language from "alive / don't break it" toward competence ("you've practiced 7 days — phrases are sticking"). The takeover copy already does some of this well; the goal-dialog copy is the laggard.

Verdict: At risk via tone. Cheapest possible fix — a copy pass on two strings — and it's arguably a bug-fix against the locked "no shaming" principle rather than a new feature. A streak-freeze / forgiveness mechanic would be a heavier, spec-worthy way to defuse the trap.

5 · Variable Reward Magnitude — handle with care

Video: Anticipation of a variable reward sustains engagement better than loss aversion (streaks).

In the code: Everything is fixed: 20/10 XP, a fixed milestone list, a deterministic daily fun-fact (app/(tabs)/index.tsx picks the fact by day index, not randomly). No anticipation loop exists.

Tension with the brand

Variable-ratio rewards are the engine of slot machines, and the app's whole identity is "warm, forgiving, not addictive." Pure variable-magnitude points would betray that. The safe reading: make the reward a variable piece of culture/competence, not a variable number. E.g. finishing a lesson reveals one of several real Karnataka cultural notes / a "you can now say…" phrase you didn't expect. The variability is in what you learn next, which serves the mission instead of exploiting it.

Verdict: Absent — and mostly should stay that way. Recommend the narrow version only (surprise cultural reveal tied to the existing fun-fact surface), explicitly not variable points. Lowest priority; easy to overdo.

6 · Completion Drive — finish the ring you started

Video: Apple Watch rings exploit the Gestalt principle of closure — an open ring nags you to close it, driving a measured 49.5% behavior change.

In the code: The app has rings but uses them for the wrong moment. The home screen shows a lesson-progress ring ("47% · 4 of 8 lessons") — good, that's closure on the curriculum. But the daily-goal ring only appears inside GoalCompleteDialog, after you've already finished. During the day, todayMinutes is tracked in stores/progressStore.ts but never shown — so there's no open ring pulling you back. You only see the ring once it's already closed, which is precisely backwards from the Watch mechanic.

Verdict: Half-built — highest ROI. Put a live daily-goal ring on home (and/or profile) that fills as todayMinutes climbs toward dailyGoalMinutes. The state is already persisted with date-rollover; this is mostly a new read + a ring component you already have a version of. This is the single most research-backed lever in the talk and it's ~80% wired.

7 · Competence vs Badge Theater — the mission-shaped gap

Video: Peloton and Chess.com retain by showing evidence of skill development (mastery), not recognition badges.

In the code: The app shows quantity and scores, not mastery. Profile surfaces "Words learned" (a cumulative count, totalPhrasesLearned) and "Overall progress %" — both are scoreboards. Games show a session score and a tier label ("Perfect!" / "Keep practising"), then forget it. Yet the per-item correctness history exists in *_progress tables, and the streak takeover already gestures at competence ("phrases are starting to feel like yours") — the vocabulary is there, the surface isn't.

Why this one matters most

The mission (SCOPE) is "belong, not pass a test." Competence evidence — "phrases you can actually use now," accuracy on greetings climbing 60%→90% — is the product promise made visible. A badge says "we recognize you." Mastery evidence says "you can do this thing you couldn't do." That's the same axis the app already sells.

Verdict: Biggest gap, best fit. Replace/augment "Words learned" with a mastery view: phrases at "learned vs getting there vs mastered" using existing per-item correctness. Heavier than #6 (needs an aggregation), but it's the truest to the brand and reuses data you're already writing. Note the locked overall-progress formula (50% lessons + 50% games, 3-game split) must stay — this would be a new mastery surface, not a redefinition of that %.

Prioritized recommendations

Ranked by (research-backed impact × fit with locked principles) ÷ effort. Every item is spec-gated: it needs a spec_docs/ doc before any code, and must clear the [LOCKED] notes called out.

  1. Live daily-goal ring on home Pattern 6

    Show an open, filling ring for todayMinutes / dailyGoalMinutes on the home screen — the Watch-ring closure lever. The state already persists with date-rollover; reuse the ring component pattern from the lesson ring / goal dialog.

    Effort: Low–Med · Touches: app/(tabs)/index.tsx, a progress selector in hooks/progress.ts · Lock check: none direct; keep it suppressed/absent when no dailyGoalMinutes is set (it's optional). Spec gate: new spec_daily_goal_ring.md.

  2. "Beat your last" in game results Pattern 2

    On each game's result screen, compare this run to the stored personal-best and surface "New best on this set!" / "1 short of your best." Self-referential, winnable, never a leaderboard.

    Effort: Med · Touches: src/games/*/components/ResultScreen.tsx, the record_*_attempt read path · Lock check: respects the locked no-leaderboard non-goal (opponent = past self). Spec gate: spec_game_personal_best.md.

  3. Mastery view replacing "Words learned" Pattern 7

    Aggregate per-item correctness into "phrases you can use now / getting there / mastered." Evidence of skill, not a count. The strongest mission fit in the list.

    Effort: Med–High (needs aggregation, maybe a server view) · Touches: app/(tabs)/profile.tsx, new accessor in services/api/ · Lock check: must NOT redefine the locked overall-progress % formula — this is a new surface alongside it. Spec gate: spec_mastery_view.md.

  4. Copy pass: streak framing from "don't lose it" → "look how far" Pattern 4

    Reword the loss-averse bits (GoalCompleteDialog "streak still alive", flame framing) toward competence. Arguably a bug-fix against the locked "no streak shaming" principle.

    Effort: Trivial · Touches: copy in GoalCompleteDialog / constants/copy.ts · Lock check: cannot remove/alter the streak mechanic (locked) — copy only. Spec gate: arguably a copy-fix exception, but confirm since it's a behavior-adjacent tone change.

  5. Decide XP's fate Pattern 1 / 3

    XP is awarded but never shown and drives no behavior. Either give it a real job or retire the field — don't let dead reward-logic accrete. Keeps net complexity flat as #1–3 add surfaces.

    Effort: Low (a decision + cleanup) · Touches: stores/progressStore.ts, total_score usage · Lock check: useProgressStore shape is [LOCKED] — removing a field needs owner sign-off + spec. Spec gate: note in an existing progress spec.

  6. Surprise cultural reveal on lesson complete Pattern 5

    The only safe variable-reward: finishing reveals one of several real Karnataka notes / an unexpected "you can now say…". Variability in content, never in points. Lowest priority, easiest to overdo.

    Effort: Med · Touches: DoneCard, fun-fact surface · Lock check: keep it non-compulsive (brand). Spec gate: spec_completion_reward.md.

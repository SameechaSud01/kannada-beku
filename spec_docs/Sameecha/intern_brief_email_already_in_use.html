Show a proper "email already in use" message

Kannada Beku · your branch: fix/auth-error-surfacing · this doc explains the bug and the path — writing the fix is your job

The bug, in one paragraph

Open the app, go to Sign up, and enter an email that already has an account. Press the button. You'll see a little message slide in saying "Couldn't sign you in — check your email and password and try again." That message is wrong: the email and password are fine. The real problem is that the account already exists — but the app never says so. Your job: make the app show an honest message like "That email already has an account — try logging in instead."

Toast — that little message that slides in and disappears by itself. This app has one shared "catalog" file that defines every toast it can show, so all messages look and behave the same.

The two files you'll touch (and no others)

  • app/(auth)/login.tsx — the login/sign-up screen. One screen handles both modes.
  • components/modals/instances/toastCatalog.ts — the toast catalog. You'll add one new toast here.

Why the bug happens

The app uses Supabase for accounts.

Supabase — the online service that stores our users. When the app calls supabase.auth.signUp(...), Supabase creates the account and replies. If something is wrong (like the email being taken), it replies with an error object instead — a bundle of info describing what went wrong.

In login.tsx, inside the function handleAuth (around lines 117–163), the sign-up code says: if Supabase returned an error, throw it. Throwing jumps straight to the catch block at the bottom:

} catch (error) {
  logger.warn('auth', 'email sign-in/sign-up failed', { err: error });
  Toasts.signInFailed();
}

Read those two lines carefully — they are the bug:

  • logger.warn(...) prints the real error to the terminal (the window on your computer where you ran the app). The developer sees it; the user never does.
  • Toasts.signInFailed() shows the same generic toast for every possible error — wrong password, no internet, email taken — all get the identical wrong message.

So Supabase does tell us "this email is taken." The catch block just throws that information away. The fix is to look at the error, recognize the "email taken" case, and show a better toast for it. Everything else should keep behaving exactly as it does now.

Step 1 — See the real error yourself

  1. Run the app. Sign up with a made-up email (e.g. test123@gmail.com) so an account exists.
  2. Sign out, then sign up with that same email again.
  3. Look at the terminal. You'll see the warning line with the error in it. If it's hard to read, temporarily add console.log(error) inside the catch — just delete it before committing.

Now look at what the error contains. It's not just text — it has named fields. The two that matter:

  • error.message — a human sentence like "User already registered".
  • error.code — a short machine label like user_already_exists.
Always check error.code, never error.message. The message is just wording — Supabase could rephrase it in any update and your check would silently stop working. The code is a stable label that won't change. Write down the exact code you see in the terminal; that's what you'll check for in Step 3.

Step 2 — Add your toast to the catalog

Open toastCatalog.ts. It's one big object where each entry is a small function. Read a few (e.g. signInFailed, resetLinkInvalid) — they all follow the same shape:

  • kind — 'error' or 'success' (controls the color).
  • id — a unique name for this toast. Must not clash with an existing one.
  • title and optional subtitle — the words the user reads.
  • optional onPress — a function to run if the user taps the toast. Look at audioFailed to see how one is passed in.

Add one new entry for "email already registered," copying the style of its neighbors. Keep the wording friendly and clear about what to do next (log in). Hint: the screen has a helper called switchMode (around line 103) that flips between login and sign-up. A toast the user can tap to jump to login mode would be a nice touch — your call.

Step 3 — Use it in the catch block

Back in login.tsx: in the catch, if the error's code is the one you wrote down in Step 1, show your new toast; otherwise leave everything exactly as it was. Keep the logger.warn line either way — we still want the error recorded.

Watch out: the same catch block also handles login failures, not just sign-up. Make sure a failed login can never show your "email already registered" toast. (Notice the isSignUp variable the function already uses.)

What NOT to do

  • Don't check whether an email exists before signing up (e.g. by querying the user list). Features that let anyone test "does this email have an account?" are a privacy problem — attackers use them to build target lists. Reacting to the error from a genuine sign-up attempt is fine; adding a lookup is not.
  • Don't touch anything else — not the Google/Apple buttons, not the password rules, not other files. Two files, small change.
  • Don't reformat or "improve" nearby code. The diff should only contain your fix.
  • Don't use any, @ts-ignore, or ! to silence TypeScript. If the error's type is unclear, look at how the rest of the file imports types from @supabase/supabase-js.

How you'll know you're done

Test every row yourself in the simulator before opening a PR:

Do thisShould happen
Sign up with an email that already has an account Your new toast. Not the old generic one.
Sign up with a brand-new email Works exactly like before — no new toast.
Log in with a wrong password Old generic toast, same as today. Your new toast must never appear here.
Check the terminal during the duplicate sign-up The logger.warn line still prints.

House rule: never say "it works" unless you've actually run it and watched it work. A short screen recording of the first test makes a great PR description.

Stuck?

  • Toast never appears → check your toast's id is unique in the catalog.
  • Can't tell which error is which → log the whole error object and look at code.
  • Anything feels ambiguous → ask before coding around it.

Issue 1 — Audio doesn't always play on word tap

Root causes (in ranked order)

  1. Shared singleton clobbered across screens. One module-level currentPlaybackSound in services/audio/deviceTtsAudioService.ts:18 is shared by every screen. Many components call stop() in their effect cleanups / unmounts (e.g. TeachWordsPhase.tsx:61-64, useGuideAudio.ts:24-28), and plays are fire-and-forget (never awaited), so one component's cleanup can silently kill the sound another component just started.
  2. Race window on rapid taps. In play() (deviceTtsAudioService.ts:53-69) the "stop previous sound" check runs before Audio.Sound.createAsync resolves and assigns the singleton. Two quick taps → the second tap sees null, doesn't stop the first, both play (or one is orphaned and leaks). There is no in-service mutex, and no screen except dictation disables the button while playing.
  3. Failures resolve silently. A sound killed by another component's cleanup does not reject, so the existing Toasts.audioFailed() never fires — the orb animates, nothing is heard, no error. Exactly the reported symptom.
  4. Deterministic bug on Emergency: app/emergency.tsx:287 passes item.audioUrl ?? item.kannada into the service; when audioUrl is set, the TTS engine literally tries to speak the URL string.
  5. No Kannada-voice guard. On a manifest miss the service falls back to device TTS without checking a kn voice exists (only the dictation game checks). Simulators usually have no Kannada voice → silence with no feedback.

Fix (Phase 1 — branch fix/audio-playback-reliability)

  1. Serialize playback with a generation token inside deviceTtsAudioService.play(): every call increments a token; after each await, a superseded call stops/unloads its own sound and exits. This closes both the double-tap race and the cross-screen clobber in one place — no per-component changes needed.
  2. Make stop() supersede too (bump the token), so an unmount cleanup can no longer kill a newer tap's sound, only whatever was current when it ran.
  3. Emergency fix: pass item.kannada to play() (drop the audioUrl fallback — no code path can fetch a remote clip anyway).
  4. Voice-availability guard: before the TTS branch speaks, check isKannadaVoiceAvailable() (already exists at deviceTtsAudioService.ts:29); if unavailable, reject so the existing Toasts.audioFailed() retry toast surfaces instead of silence.

Explicitly not doing now (flagging, not fixing — outside this bug's scope): merging the dictation game's separate audio engine (src/games/dictation/utils/audioPlayer.ts) into the main service, and the latent iOS allowsRecordingIOS reset gap (recording currently has no call sites).

Verify

  • Rapid-tap the same word 5× and alternate two words quickly — audio always plays, never doubles.
  • Tap a word, immediately navigate away and back, tap again — plays.
  • Every Emergency phrase plays spoken Kannada (not URL gibberish).
  • On the simulator (no Kannada voice) a manifest-missing word shows the retry toast instead of silence.

Issue 2 — Reset-password link fails ("Connection is unavailable") + unbranded email

The app code is complete and correct per spec_password_reset.md. The screens, PKCE exchange, and AppGate guard all exist. What's failing is the environment and dashboard config.

Why the link shows "Connection is unavailable"

requestPasswordReset (services/api/auth.ts:35) builds the redirect with Linking.createURL('reset-password'). In Expo Go that resolves to an exp://<your-Mac's-LAN-IP>:8081/--/reset-password URL — a link that only works on a device that is on the same Wi-Fi as your running Metro server, with Expo Go installed and the project open. Tapped from a computer, or from a phone when Metro isn't running / on another network, it points at an unreachable address → "Connection is unavailable." Additionally, if that exact URL isn't allow-listed in Supabase, Supabase silently redirects to the project's Site URL (typically a localhost placeholder) — same symptom. The spec itself warns: "deep links need a dev/prod build, not Expo Go" (spec_password_reset.md:212).

Fix (Phase 2 — mostly owner/dashboard, small optional code hardening)

StepWhoWhat
1OWNER Supabase → Authentication → URL Configuration → Redirect URLs: add kannada-beku://reset-password (for real builds). For continued Expo Go testing we can also log and allow-list the exact exp://…/--/reset-password value, but it will only work on the phone running Expo Go on your Wi-Fi — never on a computer.
2OWNER Supabase → Authentication → Email templates → Reset Password: set the sender name and subject to "Kannada Beku" and brand the body copy. This lives entirely in the dashboard — there is nothing in the repo to change. (I'll draft the subject/body text for you to paste in.)
3me Build an expo-dev-client development build (or use the next preview build) so the kannada-beku:// scheme actually resolves on your phone. This is the only way the end-to-end flow can genuinely work; Expo Go can never fully support it.
4both Run the spec's manual test plan (spec_password_reset.md:211-229) on the dev build: happy path, expired link, unknown email, old password rejected.

Verify

  • On a dev build: request reset → branded "Kannada Beku" email arrives → tapping the link opens the app on the reset screen → new password works, old one doesn't.

Issue 3 — "Email already in use" never shown to the user

Root cause

In app/(auth)/login.tsx:157-162, every signup error lands in one generic catch: logger.warn(...) (→ your VS Code terminal) + Toasts.signInFailed() ("Couldn't sign you in / Check your email and password") — misleading for a duplicate email. There is also no handling of Supabase's other duplicate-email signal (success-with-empty-identities, which appears if email confirmation is ever turned on).

Fix (Phase 3 — branch fix/auth-error-surfacing)

  1. Add Toasts.emailAlreadyRegistered() to components/modals/instances/toastCatalog.ts — e.g. title "That email already has an account", subtitle "Try logging in instead", with an onPress that flips the screen to login mode.
  2. In the signup path of login.tsx, map errors by Supabase error code: user_already_exists / 422 → the new toast; weak password → its own message; everything else → the existing generic toast.
  3. Defensively handle data.user.identities?.length === 0 (duplicate email under confirm-email-ON) with the same toast, replacing the current misleading confirmEmailPending() in that case.

Verify

  • Sign up with an existing email → clear "already has an account" toast, tap → login mode.
  • Sign up with a fresh email → unchanged happy path.

Issue 4 — Fake gmail accepted without checking it's real

Hard truth: no app can verify a mailbox actually exists from the client. The only real mechanism is email confirmation — send a link, and the account only becomes usable once it's clicked. Your project currently has "Confirm email" OFF as a locked decision in spec_auth_onboarding.md, so with a syntactically valid fake address Supabase happily creates the account and signs the user in — no error anywhere (the terminal errors you saw are Supabase rejecting a few addresses it can detect, swallowed by the same generic catch as Issue 3).

What I can do without reopening the locked decision (in Phase 3)

  • Tighten the client email check — the current regex /^\S+@\S+\.\S+$/ (login.tsx:44, duplicated in forgot-password.tsx:22) accepts near-garbage. Move one stricter shared validator into services/api/auth.ts (or lib/) and use it in both screens, with a specific "That doesn't look like a valid email" toast.
  • Surface Supabase's email_address_invalid error as that same toast instead of the generic one (folds into Issue 3's error mapping).

Owner decision needed DECISION

If you want typo'd/fake addresses truly caught, the fix is turning Confirm email ON in Supabase. That is a product change with real consequences: every new signup must open their inbox before using the app (more onboarding friction, and it amends a [LOCKED] decision plus needs a confirm-email deep-link/UX flow — its own small spec). My recommendation: ship the Phase 3 validation improvements now, and treat confirm-email as a separate decision for later — fake emails mainly hurt the fake-email user themselves (they can never reset their password).

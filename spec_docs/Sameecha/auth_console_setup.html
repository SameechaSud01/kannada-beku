Auth console setup

External console + dashboard prerequisites for Google login, Apple login, and password reset. This is the "step 3" companion to spec_social_login.md and spec_password_reset.md. None of this is app code — it is the configuration the code talks to. Do it before (or alongside) implementation; the features cannot be verified until it exists.

The model we chose is the native id-token flow (you build with EAS, not Expo Go). Google and Apple's native SDKs hand the app a signed identity token, which we pass to supabase.auth.signInWithIdToken(...). There is no browser redirect and no OAuth callback URL for sign-in. The only deep link in this whole guide is the password-reset link.

1 Google login iOS + Android

You create three OAuth client IDs in Google Cloud (Web, iOS, Android), then tell both the Supabase dashboard and the app about them.

1a. Google Cloud — OAuth consent screen

  1. Go to console.cloud.google.com → create or select a project (e.g. "Kannada Beku").
  2. APIs & Services → OAuth consent screen. Pick External. Fill app name, support email, developer email. Add yourself as a Test user while in "Testing" status (otherwise only you can sign in until you publish).

1b. Google Cloud — create the three clients

APIs & Services → Credentials → Create credentials → OAuth client ID, three times:

Client typeWhat to enterWhat it gives you
Web applicationName it "Supabase / web". No redirect URIs needed for native, but add the Supabase callback https://fhhzrzmmulqgmfwmeodq.supabase.co/auth/v1/callback for completeness.Web client ID + secret → both go into Supabase. The Web client ID is also the webClientId the app uses.
iOSBundle ID: com.samsud01.kannadabekuiOS client ID + its reversed form (com.googleusercontent.apps.…) used as the iOS URL scheme.
AndroidPackage: com.samsud01.kannadabeku + SHA-1 fingerprint (see 1c).Android client ID.

1c. Get the Android SHA-1 from EAS Android

The Android OAuth client needs the signing-certificate SHA-1. For EAS-managed credentials:

eas credentials
# choose: Android → (your build profile) → Keystore → "Download credentials" / view
# copy the SHA-1 Fingerprint
Two SHA-1s eventually. The EAS upload/dev keystore SHA-1 covers internal builds. If you later ship via Google Play, also add the Play App Signing SHA-1 (Play Console → your app → Setup → App integrity) as a second Android OAuth client / SHA-1, or Google sign-in breaks in production.

1d. Supabase — enable Google

  1. Supabase dashboard → Authentication → Providers → Google → toggle Enable.
  2. Client ID / Client Secret: paste the Web client's ID and secret.
  3. Authorized Client IDs (the field for native id-token validation): paste all three client IDs, comma-separated — Web, iOS, Android. This is what lets signInWithIdToken accept tokens minted by the native SDKs.
  4. If your dashboard shows a "Skip nonce check" option for Google, leave it at the default; @react-native-google-signin does not send a nonce for the basic flow, and Supabase tolerates this when the client ID is authorized.
  5. Save.
Audience must match. The id-token's audience is the Web client ID you pass to GoogleSignin.configure({ webClientId }) in the app. If sign-in fails with an "audience"/"invalid token" error, the app's webClientId and Supabase's Authorized Client IDs are out of sync.

2 Apple login iOS only

Requires a paid Apple Developer membership ($99/yr). The button is shown only on iOS; never on Android.

2a. Enable the capability on your App ID

  1. developer.apple.com → Certificates, IDs & Profiles → Identifiers.
  2. Select (or create) the App ID com.samsud01.kannadabeku.
  3. Tick Sign In with Apple in the capabilities list → Save.
For a pure native iOS sign-in, this capability plus the app.json change in §4 is often all Apple needs — the identity token's audience is your bundle ID. Steps 2b–2c (Services ID + key) are only required if you also want the web/OAuth Apple flow or if your Supabase version insists on a secret. Do 2b–2c if in doubt; it's harmless.

2b. Create a Sign in with Apple key (for the Supabase secret)

  1. Keys → + → name it, tick Sign In with Apple, configure it to your primary App ID → Continue → Register.
  2. Download the .p8 key file once (you cannot re-download it). Note the Key ID and your Team ID (top-right of the developer portal).

2c. (If using the web flow) create a Services ID

  1. Identifiers → + → Services IDs → e.g. com.samsud01.kannadabeku.web.
  2. Enable Sign In with Apple → Configure → add the Supabase callback https://fhhzrzmmulqgmfwmeodq.supabase.co/auth/v1/callback as a Return URL and your fhhzrzmmulqgmfwmeodq.supabase.co domain.

2d. Supabase — enable Apple

  1. Supabase → Authentication → Providers → Apple → Enable.
  2. Authorized Client IDs: add your bundle ID com.samsud01.kannadabeku (this is the audience of the native identity token). If you made a Services ID, add it too.
  3. If the form requires a Secret Key: Supabase will generate the client-secret JWT from your Team ID, Key ID, Services/Client ID, and the .p8 contents — paste those fields. (Native-only setups can sometimes leave this blank; fill it if the toggle won't save without it.)
  4. Save.
"Hide My Email" creates a separate account. When a user picks Apple's relay-email option, that verified address differs from any Gmail/email-password account they have, so it will not auto-link — it's a distinct account. This is expected Apple behavior (documented in spec_social_login.md), not something to fix here.

3 Password reset — redirect URL iOS + Android

The only deep link in this guide. The reset email contains a link the app must be allowed to receive.

  1. Supabase → Authentication → URL Configuration.
  2. Under Redirect URLs, add: kannada-beku://reset-password
  3. Optionally also add a wildcard for dev builds: kannada-beku://**
  4. Save.
Capture the exact value during implementation. The app computes the redirect with Linking.createURL('reset-password'). In a standalone/dev-client build this resolves to kannada-beku://reset-password, but the implementer should console.log it once and confirm it matches what you allow-listed here. (Per the [DECIDED — capture during impl] note in the reset spec.)

Email template (optional): Authentication → Email Templates → Reset Password — the default action-link macro works; rebrand the copy later if you like.

Confirm "Confirm email" stays OFF (Authentication → Providers → Email), matching the assumption the auth specs are built on.

4 app.json / env changes (the bridge — this part touches the repo)

These are config edits, allowed in app.json (not the native ios//android/ folders). They'll land with the code, but here's what they are so the picture is complete.

4a. Plugins + Apple capability in app.json

"ios": {
  "supportsTablet": true,
  "usesAppleSignIn": true,            // ← adds the Apple entitlement
  "bundleIdentifier": "com.samsud01.kannadabeku",
  "infoPlist": { /* unchanged */ }
},
"plugins": [
  "expo-router",
  "expo-secure-store",
  "expo-font",
  "@react-native-community/datetimepicker",
  "expo-notifications",
  "expo-apple-authentication",                      // ← new
  [
    "@react-native-google-signin/google-signin",
    { "iosUrlScheme": "com.googleusercontent.apps.XXXXXXXX-XXXX" }  // ← reversed iOS client ID
  ]
]

The iosUrlScheme is the reversed iOS client ID from §1b. It's a public value, so hard-coding it in app.json is fine (it is not a secret).

4b. Client IDs via env / expo-constants (locked decision)

The spec locks "client IDs come from expo-constants extra, sourced from env." Add the public client IDs to .env (and .env.example as placeholders):

# Google OAuth client IDs (public — safe as EXPO_PUBLIC_*)
EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID=xxxx.apps.googleusercontent.com
EXPO_PUBLIC_GOOGLE_IOS_CLIENT_ID=yyyy.apps.googleusercontent.com

The app reads these (e.g. GoogleSignin.configure({ webClientId, iosClientId })). Because the static app.json can't read env, the implementer may convert it to app.config.ts to inject iosUrlScheme from env — or just hard-code that one public string. That's an implementation detail, flagged in the spec.

Never put these in the app: the Google Web client secret and the Apple .p8 key. Those are real secrets and live only in the Supabase dashboard. See §6.

5 EAS dev build (you can't test any of this in Expo Go)

Native Google/Apple sign-in modules and custom-scheme deep links don't exist in Expo Go. You need a development build. Your eas.json already has a development profile (developmentClient: true) and eas-cli 20 is installed.

eas login                       # once
eas build --profile development --platform ios       # iOS dev client
eas build --profile development --platform android   # Android dev client
  • iOS: EAS will prompt to manage Apple credentials (sign in with your Apple Developer account). Register your test device first if needed: eas device:create, then install the build from the link EAS prints.
  • Android: download and install the APK from the build link. This is also where the keystore SHA-1 (from §1c) comes from.
  • After installing the dev client, run the JS: npx expo start --dev-client.
Rebuild after native changes. Adding the three native packages and the §4 plugins changes native code, so you must run a new EAS build — hot-reload won't pick them up. Rebuild once the implementer has added the deps.

Client IDs are identifiers, not credentials — Google/Apple publish them in tokens. Only the Web secret and the Apple key can authorize token exchange, so only those need to stay out of the repo.

7 Final checklist

Google

  • OAuth consent screen configured; you're a Test user
  • Web, iOS, Android OAuth clients created in Google Cloud
  • Android SHA-1 (EAS keystore) added to the Android client
  • Supabase Google provider enabled; Web ID+secret set; all 3 IDs in Authorized Client IDs

Apple iOS

  • Paid Apple Developer membership active
  • "Sign In with Apple" capability enabled on com.samsud01.kannadabeku
  • (If needed) .p8 key + Services ID created; Team ID / Key ID noted
  • Supabase Apple provider enabled; bundle ID in Authorized Client IDs; secret filled if required

Password reset

  • kannada-beku://reset-password added to Supabase Redirect URLs
  • "Confirm email" confirmed OFF

Build

  • app.json plugins + usesAppleSignIn added (lands with code)
  • Public client IDs in .env / .env.example
  • Secrets (Web secret, .p8) in Supabase only — not in repo
  • New EAS dev build installed on a real iOS device + an Android device
Suggested order. Do §3 (reset redirect URL) first — it's a one-line dashboard entry and unblocks shipping password reset, which has no other external dependency. Then Google (§1), then Apple (§2, gated on the developer account). Build (§5) last, once the implementer has added the native deps.

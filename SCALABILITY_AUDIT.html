Scalability Audit: 10 Users vs 1 Million Users

Kannada Beku · branch fix/lesson-flow-polish · 2026-07-08 (updated same day: fun facts removed & table dropped; added §2 on rings/streak/profile data; added §6 measured stress-benchmark results — finding #1 downgraded)

Verdict: Yes for the user experience — with backend caveats

The app will feel essentially identical to an individual user at 10 users and at 1 million users. The architecture is local-first: lessons 1–8, all 301 audio clips, TTS, progress, streaks, and XP live on the device. There are zero global/shared-data reads — no leaderboards, no aggregate counts, nothing that gets slower as the user base grows. Nothing on the boot path blocks on the network, and there's a hard 8-second gate past the splash even if auth hangs.

The caveat is cost and headroom on Supabase, not user-visible speed. The main lever, confirmed by a live benchmark (§6), is the ~36-request read fan-out behind the overall-progress number — measured at ~5× the wall-clock and 9× the requests of the bundled alternative, per user session. (The recompute trigger, originally ranked the top risk from static analysis, measured ~0 ms per write and is downgraded to dead-code hygiene.) At 10 users all of this is invisible; at 1M users the fan-out is the piece that pointlessly multiplies your Supabase bill. If the backend ever does get saturated, the failure mode for users is graceful for lessons but a spinner-then-error for the four games.

1 · Why the user experience holds at any scale

Per-user latency only degrades at scale if requests touch shared resources that grow with total users. This app almost entirely avoids that:

Everything the user sees on the Home rings, the streak pill, and nearly all of the Profile page is read from the local persisted Zustand store (stores/progressStore.ts via the selectors in hooks/progress.ts). None of it waits on the network, so none of it changes at 1M users:

So for this question specifically: rings, streak, and profile stats render instantly and identically at any user count, online or offline. The only per-user cost they generate server-side is one streak upsert per active day and the mastery fan-out already ranked as finding #2. The multi-device caveat from §1 applies: XP/words/minutes are local-only counters, so two devices can briefly show slightly different totals (they only ever move up, and completions/streak converge via the server).

3 · What actually scales with user count — ranked backend findings

These don't slow down any individual user directly, but they multiply per-active-user load on Postgres. At 1M registered (say 50–100k daily actives), the constant factors below dominate your Supabase bill and headroom.

#1 Dead-weight trigger: full recompute on every game tap MEASURED: negligible today

services/api/migrations/2026-05-27_db_wiring_games_and_overall.sql:272-418

AFTER INSERT/UPDATE triggers on user_lesson_progress, opposites_progress, dictation_progress, and image_match_progress call recompute_overall_progress(user_id) — a 4-way multi-CTE aggregation that scans and groups every *_items table, then UPSERTs user_overall_progress. Since the app records one attempt per answer tap, one user playing a game fires this heavy recompute on every single tap, synchronously inside the write transaction.

The app no longer reads user_overall_progress at all — overall mastery moved to a client-side rollup (services/progress/overallMastery.ts). The trigger is pure wasted work.

2026-07-08 measured correction (see §6): a live A/B benchmark — 100 writes through the triggered RPC vs 100 through the identical untriggered one — showed ~0 ms added latency per write (49.8 ms vs 49.7 ms mean; trigger verified live by the user_overall_progress row it created). The aggregates scan content tables bounded by curriculum size, not user count, so per-write cost stays roughly constant even at 1M users. This finding is therefore downgraded from top scale risk to dead-code hygiene: still worth dropping (unused output, and cost grows if content grows), but not urgent and not the dominant cost the static analysis suggested.

Fix (unchanged, urgency lowered): drop the trg_recompute_overall_progress triggers + function. One dashboard SQL run; zero app change. Verify the live trigger list first (live DB hand-patched 2026-06-13, diverges from repo migrations).

#2 Login read fan-out: ~36+ requests per session for game mastery

services/api/gameMastery.ts:71-105 · app/_layout.tsx:283 · hooks/games/*.ts:39

fetchGameMasteryByLesson issues 4 *_progress reads plus one item-fetch per game per lesson (4 games × 8 lessons ≈ 32 requests). It is prefetched on every login, and although it's cached with staleTime: Infinity, it's invalidated after every single game answer — so the next Profile visit re-runs the whole fan-out. Per-user experience is fine (it's prefetched and cached); the problem is ~36 requests/user/session × every active user, mostly re-fetching identical static content tables.

Fix (either is enough): (a) bundle the game item lists in TS like lessons already are — the fan-out collapses to 4 progress reads; or (b) replace invalidation-on-every-answer with a local cache update / invalidate-on-profile-focus, and fetch all 4 progress tables in one RPC.

#3 Global content tables served from Postgres to every device

services/api/emergency.ts:35 · services/api/games/*.ts

emergency_phrases and all four *_items tables are identical for every user but are fetched through authenticated PostgREST per device (softened by 1–24h client caches). At 1M users this is millions of redundant reads of the same few hundred rows. Lessons 1–8 already solved this the right way — bundle in the app. (Fun facts previously fell in this bucket; the feature was removed and its table dropped on 2026-07-08, which also removes its read load.)

Fix: bundle these as TS constants with the DB as an optional refresh, matching the existing TS_LESSONS / FALLBACK_GUIDE pattern. This also closes the offline gaps in finding #6.

#4 No pagination or row limits anywhere

grep: zero .range() / .limit() calls in the codebase

Safe today because content tables are tiny and per-user reads are naturally small. But fetchCompletedLessons and the four *_progress reads grow with a user's lifetime history, and any future content growth multiplies finding #2. Indexes are adequate for current per-user access patterns (PKs on (user_id, item_id), composite uniques on (lesson_id, sort_order)), so this is a discipline issue, not a current bug.

Fix: add .limit() defensively to unbounded selects; keep it as a code-review rule for new queries.

#5 Preview and production share one Supabase project

eas.json:10-37

Dev, preview, and production profiles all point at the same project and database. Preview/testing load and data mix with production. At small scale this is a data-hygiene issue; at large scale a bad preview build can degrade prod. (Correctly, only public-class keys are in the client — anon key, OAuth client IDs, Sentry DSN; no service-role key.)

Fix: create a separate free-tier Supabase project for preview/dev before user counts matter.

#6 If the backend does choke: games and Emergency are the exposed surfaces

services/api/games/*.ts · services/api/emergency.ts:37 · no timeouts in services/

This is the "breaking on user side" scenario. If Supabase is slow or down:

  • Lessons, audio, progress, guide, startup: fine (local or fallback-backed).
  • The four games are unplayable unless their 1h cache is warm — loading spinner, then error, no bundled fallback.
  • Emergency phrases — ironically advertised as the offline practice path from the Learn tab — is itself a DB read with no bundled fallback; fine once its 24h cache is warm, error on first-ever offline launch.
  • No request timeout / AbortController anywhere — a stalled (not failed) connection spins isLoading indefinitely on game screens. Only the boot path has a hard 8s gate.
Fix: bundling content (finding #3) eliminates most of this; additionally wrap Supabase reads in a ~10s Promise.race timeout so slow ≠ frozen.

#7 Minor: boot-time backfill writes, unbounded weeklyActivity keys

app/_layout.tsx:151-164 · stores/progressStore.ts:92

Two small items: (a) on launch, any local-only completed lessons are backfilled to the server via a Promise.all of RPCs — idempotent and usually empty, but it's a write on the hot boot path; (b) weeklyActivity in the persisted store adds one date key per active day and never prunes — a few KB after years of use, cosmetic only.

4 · Rough capacity picture at 1M users

Assume 1M registered → roughly 50–100k daily actives, one ~10-minute session each. Per session today: ~3 auth-time reads + ~36 mastery fan-out reads + ~20 per-tap game writes, each write currently triggering the full recompute. Streak writes are already well-designed (max once per day), and per-part lesson activity never hits the network at all.

Load sourceToday (per session)After fixes #1–#3
Reads~40 requests, mostly static content~6 requests (auth + progress only)
Writes~20 idempotent UPSERTs + ~20 heavy trigger recomputes~20 cheap idempotent UPSERTs

The write shape after fixes — small, indexed, idempotent, per-user UPSERTs — is about the friendliest workload Postgres can be given, and Supabase's pooler handles the connection side (mobile clients over PostgREST don't hold connections). With content bundled, 100k DAU is comfortable on a mid-tier Supabase compute. The §6 benchmark showed the trigger recompute is not the write-latency threat static analysis suggested (~0 ms at current content size) — the read fan-out is the real per-user cost, and it's fully removable. Also budget for Supabase Auth MAU pricing — at these numbers auth is a billing line, not a performance risk.

5 · Recommended order of work

  1. Drop the recompute_overall_progress triggers — one dashboard SQL statement, zero app change, removes the #1 cost. (Live DB diverges from repo migrations; check the live trigger list first.)
  2. Bundle game items + emergency phrases in TS (existing TS_LESSONS pattern) — collapses the login fan-out, kills the redundant global reads, and closes every offline gap in one move.
  3. Stop invalidating game-mastery on every answer — update the query cache locally or invalidate on profile focus.
  4. Add a ~10s timeout to Supabase reads so a stalled connection shows the error state instead of an endless spinner.
  5. Split preview from production Supabase before real users arrive.
  6. Housekeeping: defensive .limit() on unbounded selects; prune weeklyActivity to a rolling window.

Items 1–2 are the ones that matter for the 1M-user question; everything else is polish.

Method: two parallel code explorations covering (a) every Supabase/network call, the SQL schema, triggers, indexes, and RLS across services/, hooks/, app/, and 30+ migration files, and (b) client-side offline behavior, persisted-storage growth, the cold-start path, audio, and multi-device races. This is static analysis of the repo — no load test was run, and the live DB is known to diverge from repo migrations (hand-patched 2026-06-13), so trigger/function state should be verified in the dashboard before acting on #1.
